2200772 - FAQ: SAP HANA Statement Routing and Client Distribution Mode

SAP Note/KBA Number

Symptom

You are interested in details about SAP HANA statement routing and the client distribution mode.

Environment

SAP HANA

Cause

1. What is statement routing in SAP HANA environments?
2. How is statement routing implemented?
3. What are advantages of statement routing?
4. Which limitations exist for statement routing?
5. Are there other approaches to distribute load to the SAP HANA nodes?
6. Can connection load balancing be configured individually on client side?
7. Can statement routing be configured on SQL statement level?
8. How can I determine statement routing details for connections?
9. How can I trace the SAP HANA statement routing?
10. Is the partitioning layout considered when doing statement routing?

Resolution

1. What is statement routing in SAP HANA environments?

Statement routing is used to choose SAP HANA nodes in scale-out environments that are optimal for processing specific SQL statements. It is controlled by the client library (e.g. SQLDBC, JDBC, ODBC, DBSL) and it mainly takes into account the location of the involved tables to make a decision.

See SAP Note 2079372 and the statement routing section of SAP HANA TDI - Network Requirements for more information related to SAP HANA statement routing.

2. How is statement routing implemented?

The following picture shows how statement routing is implemented. During preparation table information is collected and sent back to the client where it is cached. The client library then chooses an optimal host (containing the requested table T1) and executes the database request on that host:

3. What are advantages of statement routing?

Statement routing minimizes the network traffic between nodes of a scale-out solution, because database requests are executed on nodes holding (parts of the) accessed tables.

4. Which limitations exist for statement routing?

The following limitations exist for statement routing:

Limitation Details
Limited to prepared statements

Table locations are determined during preparation, so statement routing only works for prepared SQL statements. If statements are not prepared, you can't take advantage of statement routing. Typical problem scenarios are:

Area SAP Note Details
SAP HANA Studio 2112849 Ad-hoc SQL statement executions in SAP HANA Studio aren't prepared and so they don't take advantage of statement routing. You can choose 'Prepare statement' in the context menu in order to force a preparation.
BO Web Intelligence 1963887 WebI 4.x doesn't send prepared statements per default. This can be changed by adjusting 'Optimize Execute' to 'False'.
Advanced Analysis for Office 2141024 AAO 1.4 SPS 11 and below may use non-prepared SQL statements. Upgrading to AAO 1.4 SPS 12 or higher fixes the problem.
Strategy when tables are located on several nodes

If accessed tables, partitions or replicas are located on several different nodes, SAP HANA follows a round-robin strategy regardless of the actual table sizes. So it can happen that a high amount of data has to be sent between the SAP HANA nodes if only a rather small table is located on the selected node.
SAP HANA bugs

The following SAP HANA bugs related to statement routing and client distribution mode exist with specific release levels:

SAP Note Rev. Details
1745057 <= 35 Connections to slave nodes may fail
1774187 <= 44 Terminations with "[7]: feature not supported: not supported to insert table info at remote nodes"
1785171 40 Terminations and crashes
2089411 <= 84 Indexserver crash
<= 122.08
<= 2.00.002.00		 Prepared statements may use outdated routing after repartitioning / redistribution

 
Configuration in table replication scenarios

As described in SAP Note 2381080, it can be useful to adjust the following parameter in case of statement routing issues in the context of table replication (SAP Note 2340450):

indexserver.ini -> [sql] -> use_new_placement_scheme_for_replicas = false
XS API dependencies

The XS engine can be used with different APIs: $.db and $.hdb. See SAP HANA XS JavaScript API Reference for more information. The newer API $.hdb has advantages like a reduced amount of metadata calls, but it doesn't use statement routing and all requests are executed on the SAP HANA master node. The older API $.db uses statement routing, but it has some other disadvantages like a higher amount of metadata calls. If you face issues like a high resource consumption on the master node caused by XS engine calls, you can consider switching to the $.db API in order to take advantage of statement routing and load distribution.

5. Are there other approaches to distribute load to the SAP HANA nodes?

Statement routing is the most popular implementation of the SAP HANA client distribution mode, but also other algorithms exist. The global client distribution mode is defined with the following parameter:

indexserver.ini -> [distribution] -> client_distribution_mode

Possible values are:

Client distribution mode Details
off No client distribution
connection

Round-robin execution of database requests

Used in the following scenarios:

  • Prepared statement: Initial connection for prepare
  • Unprepared statements
statement Statement routing based on table locations (default)
all Concurrent activation of all distribution features

6. Can connection load balancing be configured individually on client side?

Connection load balancing can be configured based on the client connection property DISTRIBUTION (see SAP Note 2079372). The default values are marked bold.

DISTRIBUTION (client SAP HANA SPS <= 10)

DISTRIBUTION (client SAP HANA SPS >= 11)

 Details
OFF OFF No client distribution
CONNECTION CONNECTION

Round-robin execution of database requests

Used in the following scenarios:

  • Prepared statement: Initial connection for prepare
  • Unprepared statements
STATEMENT_ONLY STATEMENT Statement routing based on table locations
STATEMENT ALL Concurrent activation of all distribution features

In SAP ABAP environments this setting can be implemented with the following SAP profile parameter:

dbs/hdb/connect_property = DISTRIBUTION=<distribution_mode> 

Multiple connection properties can be specified in a comma separated list, e.g.:

dbs/hdb/connect_property = DISTRIBUTION=STATEMENT, ENCRYPT=TRUE

7. Can statement routing be configured on SQL statement level?

Statement routing can be configured via SQL hints (see SAP Note 2142945):

DISTRIBUTION Details
ROUTE_BY('<table_name_1>', ..., '<table_name_n>') Route SQL statement to hosts related to the specified table(s)
ROUTE_BY_CARDINALITY('<table_name_1>', ..., '<table_name_n>') Route SQL statement to host related to the specified table with the highest amount of records
ROUTE_TO(<volume_id_1>, ..., <volume_id_n>) Route SQL statement to hosts related to the specified volume ID(s)
NO_ROUTE_TO(<volume_id_1>, ..., <volume_id_n>) Exclude routing of SQL statement to host related to the specific volume ID(s)

Starting with SAP HANA SPS 11 these hints can also be pinned to SQL statements (SAP Note 2222321).

In BW the TREXviaDBSLWithParameter function is called with a dedicated ROUTE_BY_TABLES parameter that has a similar effect like the ROUTE_BY hint.

Example:

CALL SYS.TREXviaDBSLWithParameter(  ? ,  ? ,  ? , 'ROUTE_BY_TABLES( "<schema>"."<table>" )' )

8. How can I determine statement routing details for connections?

You can use the following approaches to check for the mapping of connections to SAP HANA nodes:

  • Monitoring view M_CONNECTIONS (CONNECTION_ID, HOST)
  • SQL: "HANA_Sessions" (SAP Note 1969700)
  • SAP HANA Studio -> Administration -> Performance -> Sessions

Furthermore the following columns exist in monitoring views in this context:

Monitoring View  Column Details
M_CONNECTIONS CLIENT_DISTRIBUTION_MODE Client distribution mode (OFF, STATEMENT ROUTING, CONNECTION_DISTRIBUTION, ALL)
  LOGICAL_CONNECTION_ID Anchor connection ID (i.e. the connection used to connect to the initial node before routing is done)
M_CONNECTION_STATISTICS

EXECUTION_COUNT_BY_ROUTING

 Number of executions based on client routing
M_SQL_PLAN_CACHE

EXECUTION_COUNT_BY_ROUTING

 Number of executions based on client routing
 

IS_DISTRIBUTED_EXECUTION

 TRUE if tables are located on different nodes / services, otherwise FALSE
 

TABLE_LOCATIONS

 Locations of involved tables

9. How can I trace the SAP HANA statement routing?

You can activate a database trace or user-specific 'statementrouting' trace on 'debug' level (see SAP Note 2119087):

indexserver.ini -> [trace] -> statementrouting = 'debug'

10. Is the partitioning layout considered when doing statement routing?

As described in the SAP Online Help for Statement Routing, partitioning information can be included in the statement routing decisions. With SAP HANA SPS 12 the following options exist:

Utilization JDBC SQLDBC and related (ODBC, DBSL, ...)
Hash partitioning yes yes
Hash partitioning during split batch insert split batch not supported yes
Range partitioning no yes
Range partitioning during split batch insert split batch not supported yes
