Связь в ADSO с удалённым DTP (ППД)

09.12.2022 ~ Vyacheslav

Исправление ситуации, когда был удалён DTP из ADSO в другой объект хранилища, а связь с ним осталась. 

Если из ADSO был DTP с типом «Дельта», и в последствии он был удалён, в исходном ADSO может остаться связь, которая не позволит выполнять некоторые операции с исходным ADSO, а именно:

  1. При активации реквеста в ADSO ошибка «Для этого типа ADSO необходимо сначала распространить запрос на загрузку» (Load request must be propagated first with this type of aDSO) в ADSO с типом активация, либо АДСО типа Куб. Ошибка RSDSO_MANAGE050. 
  2. Невозможно почистить Changelog в ADSO с таблицей логов. При очистке ChangeLogа ошибка «Для этого типа ADSO необходимо сначала распространить запрос журнала изм.» (None of the selected change log requests could be removed). Ошибка RSDSO_MANAGE008.

Это связано с тем, что в некоторых версиях SAP BW (SAP BW/4HANA 1.0, SAP BW on HANA 7.40-7.50) после удаления DTP типа Delta, связь исходного инфо-провайдера остаётся с целевым. Это можно увидеть, если в администрировании ADSO нажать «Показать соединения» и там могут остаться следы удалённого ППД (но не обязательно).  В общем после удаления исходящего ППД типа Дельта, запросы присутствуют во входящей очереди АДСО, хотя они должны быть удалены. 

Решение этого вопроса описана в нотах 2481422 — Problems after deletion of outbound delta DTP with source an ADSO и 2821840 — Requests activation or data loading not working for an aDSO.

Нота 2481422 говорит о том, что отвязка удалённого ППД происходит из-за программной ошибки, и поставляет программу RSBK_RESET_DELTA_RSPM_VARIANTS, которая исправляет эти ошибки. Программа имеет единственный чекбокс, который позволяет выполнить боевой проход по исправлению во всех АДСО, либо тестовый. Указать конкретный АДСО в ней нельзя.  Программа очень простая, она обновляет (операция UPDATE) таблицу rspmvariant выбранными данными из этой же таблицы по условию, что ДТП с update-mode равен DELTA и PROCESS_TYPE равен dtpload и записи из этой таблицы не встречаются в таблице rsbkdtp или встречаются в rsbkdtpstat .

Нота 2821840 более расширенная и указывает на решение ошибок активации запросов «Для этого типа ADSO необходимо сначала распространить запрос на загрузку» (Load request must be propagated first with this type of aDSO), ошибка RSDSO_MANAGE050 и ошибку «Никакие из выбранных запросов не могут быть активирвоаны», ошибка RSDSO_MANAGE005.  Либо ошибка, когда загрузка через Delta DTP не даёт никаких записей. Это выглядит так, как будто источник данных не содержит никаких запросов, актуальная для АДСО, которые поменяли тип с Активации+ Журнал, на Оптимизированные для записи / Корпоративную память.

Суть всех этих нот сводится к установке исправляющих программ в систему. 

  1. Установка программы RSBK_RESET_DELTA_RSPM_VARIANTS, про которую написано выше. Убирает неконситентность в таблицах по удалённым ППД типа Дельта. 
  2. Программа RSPM_CHECK_VARIANT_DELTAINA, которая находит какие-то несоответсвия в таблицах.
  3. Если АДСО сконфигурировано с чейнджлогом, то ещё в этой ноте есть программа ZSAP_LIST_DELTA_RSPM_VARIANT. Она показыается список всех DTP, которые припятствуют активации запроса. 
  4. Если АДСО сконфигурировано типа куб или Data mart, то программа RSPM_ADSO_WATERMARKS, что бы показать список, который влияет на невозможность активации. Так же как было показано выше это можно посмотреть через кнопку Показать соединения (Show connections). Далее там есть кнопка «Показат ППД», по которой можно увидеть ППД в цели данных и их тип D (Delta) или F (Full).   Если ППД больше не нужен, но его и его запросы невозможно удалить, он может быть помечен для логического удаления программой RSBK_LOGICAL_DELETE_DTP. Это действие необратимо.  Если этот ППД необходим, запланируйте его ,что бы он обновлял все запросы регулярно в целе данных. 
  5. Если все вышеприведённые действия не привели к результату, надо убедиться что в этом АДСО нет красных запросов где-нибудь ниже по списку. 

Если существовала активная подписка ODP, основанная на затронутом aDSO, и эта подписка была удалена до того, как была установлена вышеуказанная нота, это может помешать процессу активации и удаления журнала изменений. Нотапомогает избежать этой ситуации в будущем, но не устраняет текущую проблему. Для решения, после установки ноты можно сделать следующее:

  1. Создайте новую подписку ODP для затронутого ADSO, запустив для него дельта-инициализацию DTP через интерфейс ODP. Параметр «Нет передачи данных; дельта-статус в источнике: не получено».
  2. После завершения этого процесса удалите подписку, удалив запрос инициализации.
